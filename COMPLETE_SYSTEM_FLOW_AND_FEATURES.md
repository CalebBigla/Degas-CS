# Complete System Flow and Features Guide

## 🎯 System Overview

This is a **Dynamic Attendance and Onboarding Management System** designed for organizations like churches, schools, conferences, and businesses to manage user registration and track attendance efficiently.

### Core Concept
- **Register Once** → Users create account one time
- **Attend Multiple Times** → Users mark attendance at every event
- **Two QR Systems** → Personal QR codes + Location QR codes
- **Dynamic Forms** → Create unlimited registration forms for different user groups
- **Automatic Tables** → Database tables created automatically from forms

---

## 🏗️ System Architecture

### Technology Stack
- **Frontend**: React + TypeScript + Vite + TailwindCSS
- **Backend**: Node.js + Express + TypeScript
- **Database**: SQLite (local) / PostgreSQL (production)
- **Authentication**: JWT tokens (dual system)
- **QR Codes**: QRCode.js library with encryption

### Database Structure
```
core_users              → User accounts (email, password)
user_data_links         → Links users to their data tables
form_definitions        → Form metadata
form_fields             → Form field definitions
attendance_sessions     → Event sessions
attendance_records      → Attendance tracking
qr_codes               → QR code storage
Dynamic Tables         → Members, Students, Staff, etc.
```

### 4. Two QR Code Systems

#### Type 1: User QR Code (Personal)
**Purpose**: Identity verification and manual check-in

**Features**:
- Unique per user
- Contains encrypted user UUID + table name
- Generated at registration
- Downloadable as PNG
- Viewable in user dashboard
- Never expires
- Used by admin to scan user

**Format**: `{uuid: "xxx", table: "Members"}` (encrypted + base64)

**Use Cases**:
- Security checkpoints
- VIP entry verification
- Manual attendance by admin
- Identity verification
- Access control


#### Type 2: Location QR Code (Event)
**Purpose**: Self-service attendance marking

**Features**:
- Unique per session
- Contains session ID + timestamp + signature
- Generated by admin
- Reusable for recurring events
- Expires when session ends
- Displayed at entrance/location
- Users scan to check in

**Format**: `SESSION:{sessionId}:{timestamp}:{signature}` (base64)

**Use Cases**:
- Church services (weekly)
- School classes (daily)
- Conference sessions
- Event check-ins
- Self-service attendance

## 🔄 DETAILED SYSTEM FLOWS

### Flow 1: Admin Setup (First Time)

```
1. Admin Login
   ↓
2. Navigate to Forms Page (/admin/forms)
   ↓
3. Click "Create Form"
   ↓
4. Fill Form Details:
   - Form Name: "Church Membership 2026"
   - Target Table: "Members"
   ↓
5. Add Fields:
   - Name (text, required)
   - Email (email, required, ✓ Email Field)
   - Password (password, required, ✓ Password Field)
   - Phone (tel, optional)
   - Address (textarea, optional)
   - Photo (file, optional)
   ↓
6. Click "Create Form"
   ↓
7. System Actions:
   - Validates form (must have email + password fields)
   - Creates form_definition record
   - Creates form_fields records
   - Creates "Members" table in database
   - Generates form ID
   ↓
8. Form Created Successfully
   ↓
9. Copy Registration Link
   - Format: http://localhost:5173/register/{formId}
   ↓
10. Share Link with Users
```

### Flow 2: User Registration

```
1. User Receives Registration Link
   ↓
2. Opens Link in Browser
   - URL: /register/{formId}
   ↓
3. System Loads Form
   - Fetches form definition from API
   - Renders fields dynamically
   ↓
4. User Fills Form:
   - Name: "John Doe"
   - Email: "john@example.com"
   - Password: "secure123"
   - Phone: "1234567890"
   - Uploads photo (optional)
   ↓
5. User Clicks "Register"
   ↓
6. Frontend Validation:
   - Required fields filled
   - Email format valid
   - Password strength check
   ↓
7. Submit to Backend
   - POST /api/onboarding/register/{formId}
   ↓
8. Backend Processing:
   a. Validate form data
   b. Check email doesn't exist
   c. Hash password
   d. Create core_users record
   e. Generate user UUID
   f. Insert into Members table
   g. Create user_data_links record
   h. Generate User QR code
   i. Generate JWT token
   ↓
9. Return Success Response:
   - User ID
   - Email
   - QR Code (base64 image)
   - JWT Token
   ↓
10. Frontend Shows Success Screen:
    - "Registration Successful!"
    - Display QR Code
    - "Download QR Code" button
    - "Go to Dashboard" button
    ↓
11. User Auto-logged In
    - Token stored in localStorage
    - Redirected to /my-dashboard
```

### Flow 3: Create Attendance Session

```
1. Admin Logs In
   ↓
2. Navigate to Attendance Page (/admin/attendance)
   ↓
3. Click "Create Session"
   ↓
4. Fill Session Details:
   - Session Name: "Sunday Service - March 30, 2026"
   - Description: "Weekly worship service"
   - Start Time: 2026-03-30 10:00 AM
   - End Time: 2026-03-30 12:00 PM
   - Grace Period: 15 minutes
   ↓
5. Click "Create"
   ↓
6. Backend Processing:
   - Validate session data
   - Create attendance_sessions record
   - Generate session ID
   - Set is_active = true
   ↓
7. Session Created
   ↓
8. Admin Clicks "Generate QR Code"
   ↓
9. Backend Generates Location QR:
   - Format: SESSION:{sessionId}:{timestamp}:{signature}
   - Encode to base64
   - Generate QR image
   - Store in session record
   ↓
10. Display QR Code
    - Show QR image
    - "Download QR Code" button
    - "Print" button
    ↓
11. Admin Downloads/Prints QR
    ↓
12. Admin Displays QR at Entrance
    - Print poster
    - Display on TV/monitor
    - Place at registration desk
```
